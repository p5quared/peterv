If you develop software for a living, even if you haven't read
[Manifesto for Agile Software Development](https://agilemanifesto.org/),
you're probably familiar with the term. Agile is not a system so much as
it is a philosophy for building software with principles like being
_reactive_ to change rather than attempting to predict everything and
_collaborating_ with stakeholders rather than negotiating.

Its success is that teams using Agile can build _working_ 
software fast, because _working_ is a definition
which evolves in digestible pieces. Before Agile, teams would
be more likely to spend vast amounts of time mapping out every feature,
from _what_ it will do, its API to users, and _how_ the software would
enable that. Then engineers would go into a cave (metaphorically) and 
build the system to spec. Naturally by the time they finished
(or before they were finished), the spec would be out of date 
and chunks of their work was invalidated.

The reason I'm writing this is because I wonder, if agentic development
is the revolution we are claiming, then maybe we need to rethink
the development process to accommodate this Brave New World.

## Agile for Agents

Let's start by discussing what makes development _different_ with agents.
I'm going to start discussing what makes development with agents _different_ 
by making some sweeping assumptions about what _optimal_ agentic 
development looks lke. _Agents can make the cost, time commitment, of writing
code 0. Agents cannot make the cost of decision making 0._
